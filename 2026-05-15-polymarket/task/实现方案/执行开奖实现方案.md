# 执行开奖实现方案

## 目标
* 在 `doge-task` 中新增“执行开奖”定时任务
* 以 `oracle_draw` 作为开奖主驱动表执行自动开奖
* 对 `oracle_draw.status = 1` 且关联 `oracle_issue.status = 3` 的记录执行自动开奖
* 更新开奖记录 `oracle_draw`
* 生成收益记录 `oracle_income`
* 回写订单状态 `oracle_order`
* 最终将期次状态更新为 `已开奖`

## 表结构确认
* `oracle_income` 需要新增应发字段,其余表结构无需调整
* `oracle_draw.status`: 1=待开奖,2=已开奖,3=数据待补,4=结算中
* `oracle_order.status`: 1=待开奖,2=已结算,3=未中奖
* `oracle_income.bill_status`: 1=已发放,2=未发放,3=不处理
* `oracle_income.should_income_amount`: 理论应发收益金额
* `oracle_income.should_profit_amount`: 理论应发盈利金额

### 追加字段 SQL
```sql
ALTER TABLE oracle_income
ADD COLUMN should_income_amount DECIMAL(32,18) NOT NULL DEFAULT 0 COMMENT '应发收益金额',
ADD COLUMN should_profit_amount DECIMAL(32,18) NOT NULL DEFAULT 0 COMMENT '应发盈利金额';
```

## 方案边界
* 本任务只负责自动开奖、更新开奖记录、生成收益记录
* 本任务负责同步维护 `account_oracle` 的结算汇总字段
* 本任务不直接发放资产,仅生成 `oracle_income`
* 资产发放建议拆分为独立任务处理
* 自动开奖失败后不做无限重试,异常期次交给后台“补数据/重试开奖”处理

## 前置约束
* “进行中 -> 待开奖”的补充任务会预创建 `oracle_draw`
* 执行开奖时不能再以 `oracle_issue` 作为主驱动表,而应直接处理待开奖的 `oracle_draw`
* 自动开奖必须依赖 `open_price` 与 `close_price`
* 若 `oracle_draw.open_price` 或 `oracle_draw.close_price` 缺失,可尝试从 `oracle_issue` 补齐
* 若补齐后仍缺失,本次不能继续自动开奖,需要转入“数据待补”
* 业务日志需覆盖开奖上下文加载、价格来源、订单统计、分支判断、状态回写和收益生成汇总

## 定时任务设计
### 调度时间
* `0 5 * * * ?`

### 锁设计
* 任务入口锁: `lock:handle:oracle:draw:execute`
* 单记录锁: `lock:handle:oracle:draw:execute:{drawId}`
* 必须同时保留“任务入口锁 + 单条数据锁”,与现有 `doge-task` 规范保持一致

## 执行流程
1. 定时任务查询 `oracle_draw.status = 1` 的开奖记录,按 `id asc` 处理
2. 每条开奖记录单独加锁后进入处理方法
3. 再次查询开奖记录,二次校验 `status = 1`
4. 根据 `draw.issue_id` 查询关联期次 `oracle_issue`
5. 若关联期次不存在,将 `oracle_draw.status` 更新为 `3=数据待补`,备注“关联期次不存在”
6. 若关联期次存在但 `oracle_issue.status != 3`,说明状态不一致,记录日志后跳过本次处理
7. 校验 `oracle_draw.open_price`、`oracle_draw.close_price` 是否完整
8. 若开奖记录价格缺失,尝试从 `oracle_issue.open_price`、`oracle_issue.close_price` 补齐
9. 若补齐后仍缺失,将 `oracle_draw.status` 更新为 `3=数据待补`,备注写清缺失原因,期次仍保持 `status = 3`
10. 查询本期 `oracle_order.status = 1` 的订单列表
11. 汇总订单总金额、手续费总金额、有效金额总计、正方人数、正方有效金额、反方人数、反方有效金额、订单数
12. 根据公式 `close_price >= open_price ? 1 : 2` 计算开奖结果
13. 计算胜方有效金额
14. 若 `total_effective_amount <= 0`,说明整期无人参与,直接完成开奖,不生成收益记录
15. 若 `total_effective_amount > 0` 且胜方有效金额小于等于0,则查询 `sys_config` 中 `ORACLE_MARKET_ASSET_POOL_ACCOUNT_ID`
16. 若系统资金池账户未配置或用户不存在,则将 `oracle_draw.status` 更新为 `3`,备注“系统资金池账户未配置或不存在”
17. 若系统资金池账户存在,则将所有真实用户订单结算为未命中,并额外生成一条归属于系统资金池账户的 `oracle_income`
    * 该条系统资金池收益记录没有真实用户订单,`order_id` 固定写 `0` 作为占位,`order_no` 复用 `draw_no`
18. 使用事务统一处理开奖记录、订单状态、收益记录写入
19. 事务开始后先将当前 `oracle_draw.status` 更新为 `4=结算中`
20. 常规场景下先预计算每笔订单的 `should_income_amount`、`should_profit_amount`、`income_amount`、`profit_amount`
21. 命中订单 `income_amount`、`profit_amount` 按两位小数向下取整落库
22. 未命中订单 `should_income_amount = 0`、`income_amount = 0`
23. 汇总所有命中订单实发收益,计算 `尾差 = total_effective_amount - sum(income_amount)`
24. 若尾差大于0,则额外生成一条归属于系统资金池账户的尾差收益记录
25. 命中订单更新 `oracle_order.status = 2`
26. 未命中订单更新 `oracle_order.status = 3`
27. 在生成 `oracle_income` 的同时,按账户汇总更新 `account_oracle.total_income_amount`、`account_oracle.total_profit_amount`
28. 若账户汇总不存在,则补建 `account_oracle` 记录后再累加
29. 所有收益记录生成完成后,更新 `oracle_draw.status = 2`,并回写 `draw_result`、`drawn_time`、`income_num`、统计字段
30. 最后更新 `oracle_issue.status = 4`

## 收益计算口径
* 净清算池 = 本期 `total_effective_amount`
* 胜方有效金额 = `positive_amount` 或 `negative_amount`
* 命中订单占比 = 当前订单 `effective_amount / 胜方有效金额`
* 理论应发收益 `should_income_amount` = `净清算池 * 命中订单占比`
* 理论应发盈利 `should_profit_amount` = `should_income_amount - amount`
* 实发收益 `income_amount` = `should_income_amount` 向下保留两位小数
* 实发盈利 `profit_amount` = `income_amount - amount`
* 未命中订单 `should_income_amount = 0`
* 未命中订单 `should_profit_amount = 0 - amount`
* 未命中订单 `income_amount = 0`
* 未命中订单 `profit_amount = 0 - amount`
* 系统尾差 `should_income_amount = total_effective_amount - sum(should_income_amount)`
* 系统尾差 `income_amount = total_effective_amount - sum(命中订单 income_amount)`
* `oracle_income.positive_rate` 建议复用为“中奖占比”
* 命中订单 `oracle_income.status = 1`, `bill_status = 2`
* 未命中订单 `oracle_income.status = 2`, `bill_status = 3`
* 若胜方无人命中,则系统资金池账户补写一条 `oracle_income.status = 1`, `bill_status = 2` 的归集收益记录
* 若存在系统尾差,则系统资金池账户额外补写一条尾差收益记录

## account_oracle 维护口径
* `account_oracle.total_pay_amount`、`total_fee_amount` 继续由支付环节维护
* `account_oracle.total_income_amount`、`total_profit_amount` 在本任务生成 `oracle_income` 时同步维护
* 汇总口径统一使用实发字段 `income_amount`、`profit_amount`
* 命中订单:
  `total_income_amount += income_amount`
  `total_profit_amount += profit_amount`
* 未命中订单:
  `total_income_amount += 0`
  `total_profit_amount += (0 - amount)`
* 系统资金池账户归集收益:
  `total_income_amount += total_effective_amount`
  `total_profit_amount += total_effective_amount`
* 系统资金池账户尾差收益:
  `total_income_amount += 余数 income_amount`
  `total_profit_amount += 余数 profit_amount`
* 维护原则:
  `account_oracle` 是结算汇总,不是已发放汇总,因此应在“执行开奖”落库 `oracle_income` 时处理,不应放到收益发放任务中处理

## 异常兜底
### 数据缺失
* 关联期次不存在
* `open_price` 缺失
* `close_price` 缺失
* 系统资金池账户未配置
* 系统资金池账户不存在
* 订单统计数据异常

### 兜底处理
* 写入 `oracle_draw.status = 3`
* `remark` 明确记录异常原因
* `oracle_issue.status` 保持 `3=待开奖`
* 不自动生成 `oracle_income`
* 后续由后台补数据后触发“重试开奖”

## 事务建议
* 以下动作必须放在同一个事务里
* 插入 `oracle_income`
* 插入或更新 `account_oracle`
* 更新 `oracle_order`
* 更新 `oracle_draw`
* 更新 `oracle_issue`

## SQL建议
```sql
SELECT
    id,
    draw_no,
    issue_id,
    issue_code,
    open_price,
    close_price,
    status
FROM
    oracle_draw
WHERE
    status = 1
ORDER BY
    id ASC;
```

```sql
SELECT
    id,
    issue_code,
    open_price,
    close_price,
    status
FROM
    oracle_issue
WHERE
    id = #{issueId};
```

```sql
SELECT
    id,
    order_no,
    account_id,
    account_no,
    issue_id,
    issue_code,
    issue_market_config_id,
    market_config_name,
    symbol,
    choice_value,
    amount,
    fee,
    effective_amount,
    status
FROM
    oracle_order
WHERE
    issue_id = #{issueId}
    AND status = 1
ORDER BY
    id ASC;
```
